# 橋接模式 (Bridge Pattern)

1. 橋接模式的核心定義
    **核心精神：** 不僅允許你改變實作（Implementation），也允許你改變抽象（Abstraction），將兩者放在各自獨立的類別階層中，讓它們可以獨立地變化。

    為什麼我們需要它？（問題場景）
        想像一下你正在為電視開發一款符合人體工學、對使用者友善的「萬用遙控器」。
        1. **實作面（電視機）會變：** 你的客戶會擁有各種不同品牌、型號的電視（例如 Sony、RCA）。
        2. **抽象面（遙控器）會變：** 遙控器的使用者介面不會一次到位，隨著收集到更多的使用者回饋，遙控器的介面與功能會不斷優化與改版。

    如果我們使用傳統的繼承，把遙控器和電視機綁死在一起，很快就會面臨恐怖的「類別爆炸」（例如：`Sony新版遙控器`、`Sony舊版遙控器`、`RCA新版遙控器`...）。

2. 橋接模式的架構與流程圖

    為了同時讓「遙控器」和「電視機」能各自獨立進化，我們將它們拆分成兩個平行的類別階層，並在中間建立一座「橋（Bridge）」（透過 `HAS-A` 組合關係）。

    ```text
           [ 抽象層階層 (Abstraction) ]              [ 實作層階層 (Implementation) ]
    
                   遙控器抽象概念                                 電視機實作介面
                [ RemoteControl ] ------------------------> [      TV       ]
                + on()            |     (這就是 Bridge)      + on()          |
                + off()           |     HAS-A implementor   + off()         |
                + setChannel()    |                         + tuneChannel() |
                       ^                                            ^
                       | 繼承/擴充                                   | 實作
                       |                                            |
             +---------+---------+                        +---------+---------+
             |                   |                        |                   |
       [ 新一代遙控器 ]      [ 舊版遙控器 ]               [ Sony TV ]       [ RCA TV ]
     (ConcreteRemote)                             (Concrete Impl)   (Concrete Impl)
     + nextChannel()                              + tuneChannel()   + tuneChannel()
     + previousChannel()
    ```

    在這個架構中：
    * **抽象層（RemoteControl）：** 擁有一個指向實作層（TV）的參考變數（也就是 Bridge）。抽象層裡的所有方法，都是透過呼叫實作層的方法來完成的。
    * **具體抽象層（ConcreteRemote）：** 是基於抽象層去擴充的（例如加入上一台、下一台的功能），它只需要呼叫自己抽象層的方法，而不需要知道底層是哪牌的電視。



3. 總結

    **主要優點 (Benefits)：**
    1. **完美的解耦：** 將實作解耦，使得實作不會永久被綁死在單一個介面上。
    2. **雙向自由擴充：** 抽象層（前端介面）與實作層（底層 API）可以各自獨立擴充，互不干擾。
    3. **隔離變動：** 更改具體抽象類別的細節，完全不會影響到使用它的客戶端。
    4. **跨平台開發首選：** 任何時候當你需要讓介面和實作以不同的方式變化時（例如：需要在多個不同平台上運行的圖形和視窗系統），這都是非常實用的模式。
    
    **主要缺點 (Drawbacks)：**
    1. **增加系統複雜度：** 對於初學者或接手維護的工程師來說，需要同時理解與追蹤兩個平行的類別階層，這增加了設計與閱讀程式碼的複雜性。在決定使用前，必須先衡量未來的擴充需求是否值得引入這樣的複雜度。

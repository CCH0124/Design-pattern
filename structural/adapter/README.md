# Adapter Pattern

## 核心目的與定義

轉接器模式（Adapter Pattern）旨在將一個類別的介面轉換成客戶端期望的另一種介面。它允許原本因介面不相容而無法協同工作的類別能夠順利合作。

* **定義**：轉接器模式將一個類別的介面轉換為客戶端所期望的另一個介面。轉接器讓原本因介面不相容而無法協同工作的類別能夠協同運作。
* **角色職責**：轉接器（Adapter）實作了客戶端預期的**目標介面**（Target Interface），同時持有**供應商介面**（Adaptee Interface）的實例，並將客戶端的請求轉換為對供應商類別的調用。

## 實作細節與類型

1. **物件轉接器 (Object Adapter)**：這是 Java 中常見的實作方式。轉接器使用**組合**（composition）來包裹供應商物件，並將請求委派（delegates）給被包裹的物件。這種方法允許轉接器與供應商類別的任何子類別協同運作。
2. **類別轉接器 (Class Adapter)**：這需要多重繼承，轉接器必須同時繼承目標介面和供應商類別。由於 Java 不支持多重繼承，此類型不適用。

## 應用案例

* **鴨子與火雞 (Duck and Turkey)**：在模擬器中，火雞（`Turkey`）是供應商（Adaptee），它會「呱呱叫」（`gobble`）和短距離飛行；而模擬器期望看到鴨子（`Duck`，Target），它會「叫」（`quack`）和長距離飛行。`TurkeyAdapter` 實作了 `Duck` 介面，當客戶端調用 `quack()` 時，轉接器實際調用火雞的 `gobble()` 方法。
* **枚舉器與迭代器 (Enumeration and Iterator)**：轉接器可用於將舊的 `Enumeration` 介面（供應商）轉換為新的 `Iterator` 介面（目標）。由於 `Enumeration` 不包含 `remove()` 方法，因此轉接器實作 `Iterator.remove()` 時，應拋出 `UnsupportedOperationException` 異常。

## 與其他模式的關係

* **與裝飾者模式 (Decorator) 的區別**：兩者都涉及包裹物件。但轉接器模式的**意圖**是改變介面，而裝飾者模式的意圖是增加新的行為和職責，但不改變介面。
* **與代理模式 (Proxy) 的區別**：轉接器改變了被包裹物件的介面，而代理模式則實作與其實體物件相同的介面，其目的是控制存取。

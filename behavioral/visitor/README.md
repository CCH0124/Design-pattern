# 訪問者模式 (Visitor Pattern)

在處理複雜的階層或樹狀結構資料（例如：系統拓撲圖、設定檔樹狀解析、編譯器的抽象語法樹 AST）時，我們經常會遇到一個難題：**如何在不修改現有資料結構類別的前提下，為這些資料新增各式各樣的操作與計算能力？**


1. 訪問者模式的核心概念與問題場景

   **定義：** 當你想要為一個由許多物件組成的複合結構（Composite）加入新的能力，且「不在乎破壞物件的封裝性」時，就可以使用訪問者模式。
   
   **痛點場景：**
   想像我們用「複合模式 (Composite Pattern)」設計了一套餐廳菜單系統，裡面包含了菜單 (Menu)、子菜單、餐點 (MenuItem) 以及食材 (Ingredient)。
   現在，因為健康意識抬頭，系統需要新增取得「營養資訊」的功能（包含計算卡路里、蛋白質、碳水化合物等）。
   如果我們在 `Menu`、`MenuItem` 和 `Ingredient` 每個類別裡都加上 `getCalories()`、`getProtein()` 等方法，當未來又要增加「食譜輸出」或「成本分析」時，我們就得把所有類別打開再改一遍，這是一場牽一髮動全身的維護災難。
   
   **解決方案：**
   訪問者模式的核心思想是：**將「資料結構」與「對資料的操作邏輯」徹底分離**。它會將新的操作與計算封裝在一個獨立的「訪問者 (Visitor)」物件中，這樣複合結構的類別就完全不用修改。


### 2. 系統架構與運作流程圖

   在訪問者模式中，我們通常會搭配一個**走訪器 (Traverser)**。走訪器知道如何導覽整個複合結構的每一個節點，並在走訪過程中，引引導**訪問者 (Visitor)** 去收集每個節點的狀態。

   ```text
          (1) 指揮走訪器開始導覽結構
   [ Client (客戶端) ] ------------------------+
                                               |
                                               v
   [ Traverser (走訪器) ] ----------------------------------------+
      | (2) 負責導覽 Composite 結構，並引導 Visitor 拜訪每個節點     |
      v                                                           |
   [ Visitor (訪問者) ]                                           |
      |                                                           |
      | (3) 呼叫節點的 getState() 收集狀態                        |
      +---------------------------------------------------+       |
      | (4) 內部執行新操作 (例如: getCalories, getProtein)  |       |
      v                                                   v       v
   [ 新功能的計算結果 ]                           [ Composite 結構 (Menu/MenuItem 等) ]
                                                + getState() : 僅負責暴露自身狀態
   ```

   在這個流程中，原本的複合節點（如 Menu 或 MenuItem）不需要實作複雜的營養計算邏輯，它們只需要提供一個 `getState()` 方法，將自己的內部資料（狀態）交給 Visitor 即可。當你需要新增任何功能時，只要寫一個新的 Visitor 類別就好了。

3. 總結

   在大型系統架構中引入 Visitor 模式是一個重大的決定，因為它帶來了極高的操作擴充性，卻也犧牲了物件導向的核心原則。

   **主要優點 (Benefits)：**
     * **符合開閉原則 (Open/Closed Principle)：** 允許你在不修改現有複合結構程式碼的情況下，輕鬆加入新的操作或功能。
     * **高度的邏輯集中 (Cohesion)：** 將某個特定功能（例如營養計算）的所有邏輯，統一集中在單一個 Visitor 類別中，而不是散落在各個資料類別裡，大幅提升了程式碼的集中度與可維護性。

   **主要缺點與系統風險 (Drawbacks)：**
     * **破壞封裝性 (Encapsulation is broken)：** 這是最關注的代價。為了讓 Visitor 能夠進行計算，原本的複合結構物件必須把內部的狀態（資料）透過 `getState()` 完全暴露出來。
     * **難以應付結構本身的變動：** 因為 Traverser 與 Visitor 緊密依賴著複合結構的節點類型，如果你的系統經常需要新增「新的節點類型」（例如新增一種特殊的菜單類別），那麼所有的 Traverser 與 Visitor 都必須跟著修改，這會變得非常困難。


   如果你面對的系統，其**「資料結構非常穩定，幾乎不會增加新節點」**，但**「需要頻繁地為這些資料增加新的分析或操作功能」**，且你可以接受稍微犧牲封裝性，那麼訪問者模式 (Visitor) 會是讓程式碼保持乾淨俐落的殺手級模式。

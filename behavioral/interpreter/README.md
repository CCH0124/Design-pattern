# 解譯器模式 (Interpreter Pattern)

1. 解譯器模式的核心定義

   **定義：** 使用解譯器模式來為某種語言建立解譯器 (Interpreter)。
   當你需要實作一個簡單的語言時，這個模式會為該語言的語法定義一種「基於類別 (class-based) 的表示方式」，並同時提供一個解譯器來直譯這些語句。

   簡單來說，它的核心精神就是：**把每一條語法規則，都轉化為程式中的一個「類別」**。

   流程圖輔助理解:

   在解譯器模式中，會定義一個共同的 `Expression` (表達式) 介面，所有語法結構都會實作這個介面，並透過遞迴呼叫 `interpret(context)` 來執行指令：
   
   ```text
          [ 待解析的程式碼與狀態 (Context) ]
                          |
                          v
       [ 抽象表達式介面 (Expression) : interpret(context) ]
                          | (實作不同語法規則的類別)
         +----------------+----------------+----------------+
         |                |                |                |
   [ 迴圈/語法控制 ] [ 循序/語法組合 ]      [ 變數 ]       [ 具體命令 ]
    (Repetition)      (Sequence)       (Variable)   (FlyCommand 等)
         |                |                |                |
         +----------------+----------------+----------------+
                          | 執行 interpret(context) 方法
                          v
                 [ 改變系統狀態或輸出執行結果 ]
   ```

2. 優缺點與應用評估

   在架構設計上，是否要自己刻一套解譯器，需要進行權衡。根據書中整理的特點如下：

   主要優點 (Benefits):

   * **極易實作與擴充：** 因為將每一條語法規則都表示為一個獨立的類別，所以語言變得非常容易實作。同時，由於語法是由類別所呈現的，你可以很輕易地修改或擴充這套語言。
   * **彈性附加新行為：** 透過在這個類別結構中加入新方法，你可以輕鬆加上「解譯」以外的新行為，例如：程式碼美化排版 (pretty printing) 或是更複雜的程式驗證邏輯。

   應用場景與致命傷 (Uses and Drawbacks)

   * **適用情境：** 這個模式非常適合用來實作簡單的語言、腳本或程式語言。當你面對一個非常單純的語法，而且在設計上「簡單性」比「執行效率」更重要時，這就是很好的選擇。
   * **系統致命傷：** 作為工程師必須注意，如果這個語言的文法規則數量變得非常龐大，這個模式的類別數量會暴增，變得極度笨重且難以維護。在這種情況下，改用成熟的語法解析器或編譯器產生器 (parser/compiler generator) 會是更正確的架構選擇。

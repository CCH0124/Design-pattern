# 中介者模式 (Mediator Pattern)

在設計複雜系統（例如微服務架構或大型應用程式）時，我們經常會遇到一個問題：系統內的各個元件需要互相溝通，若讓它們直接對話，最後會演變成宛如蜘蛛網般錯綜複雜的「網狀相依關係」。

為了解決這種高度耦合的噩夢，**中介者模式 (Mediator Pattern)** 就派上用場了。它的核心目的非常明確：**將相關物件之間複雜的溝通與控制集中管理**。

1. 中介者模式的核心概念

   想像一下飛機與塔台的關係。如果每架飛機都要互相廣播來確認航線，空中交通肯定會大亂。因此，所有飛機只跟「塔台（中介者）」溝通，由塔台來統一負責協調與控制。

   在軟體系統中，中介者模式引入了一個 Mediator 物件，所有的實體物件（或元件）不再直接互相呼叫，而是統一將狀態改變通知中介者，並由中介者來決定接下來要觸發哪些物件的行為。

   流程圖輔助理解:

   **【導入前：高耦合的蜘蛛網】**
      各物件必須互相認識並呼叫彼此，牽一髮動全身。
      
      ```text
       [ 鬧鐘 ] <--------> [ 咖啡機 ]
          ^  \             /  ^
          |   \           /   |
          v    v         v    v
      [ 行事曆 ] <--------> [ 灑水器 ]
      ```
   
   **【導入後：以中介者為中心的星狀架構】**
      物件彼此完全解耦，只單向與中介者溝通。
      
      ```text
       [ 鬧鐘 ]          [ 咖啡機 ]
             \            /
              v          v
            [ Mediator (中介者) ]
              ^          ^
             /            \
       [ 行事曆 ]         [ 灑水器 ]
      ```


2. 總結

   在架構設計上，引入中介者模式有明顯的利弊，我們必須謹慎評估：

   **主要優點 (Benefits)：**
      * **徹底解耦 (Decoupling)：** 將物件從系統中解耦，大幅提升了個別物件的「可重用性 (Reusability)」。
      * **集中維護 (Centralization)：** 將複雜的控制邏輯集中在單一位置，系統的維護與修改變得更簡單。
      * **簡化訊息傳遞：** 減少並簡化了系統內物件之間互相傳遞的訊息種類。
      * **實務應用：** 在前端或客戶端開發中，中介者模式非常常用於協調複雜的 GUI (圖形使用者介面) 元件。

   **主要缺點與風險 (Drawbacks)：**
      * **中介者過度膨脹 (God Object 反模式)：** 這是架構師最怕看到的狀況。如果設計不當，這個中介者物件會吸收所有的邏輯，最後變得過度複雜、龐大且難以維護。

   總結來說，當你的系統出現了大量的元件，且它們之間的溝通邏輯已經複雜到讓你難以追蹤時，抽出一個「中介者」來做統一的指揮與調度，將會是釐清系統架構的絕佳手段。

   下面是核心 Java 的一些示例：

   * java.util.Timer （所有 scheduleXXX()方法）
   * java.util.concurrent.Executor#execute()
   * java.util.concurrent.ExecutorService （ invokeXXX()和 submit()方法）
   * java.util.concurrent.ScheduledExecutorService （所有 scheduleXXX()方法）
   * java.lang.reflect.Method#invoke()

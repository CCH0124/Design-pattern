# 策略模式 (Strategy Pattern)

在開發高併發系統、維護底層基礎架構或設計複雜演算法時，我們最常遇到的挑戰就是「需求與商業邏輯不斷改變」。

當一個系統需要支援多種不同的演算法或行為（例如：不同的負載平衡路由策略、不同的資料壓縮格式，或是不同的定價計算邏輯）時，如果我們將所有邏輯硬塞在同一個類別裡，並使用龐大的 `if-else` 或 `switch` 判斷式來切換，系統將會變得難以維護且極度脆弱。

為了解決這個問題，**策略模式 (Strategy Pattern)** 提供了非常優雅的底層架構。

1. 策略模式的核心概念

    **定義：** 策略模式定義了一個演算法家族，將每一個演算法分別封裝起來，並讓它們可以互相替換。策略模式讓演算法的變動可以獨立於使用該演算法的客戶端之外。

    這個模式完美體現了兩個極為重要的物件導向與系統設計原則：
    * **針對介面（超型態）寫程式，而不是針對實作寫程式**：讓主體只依賴抽象的行為介面。
    * **多用合成 (Composition)，少用繼承 (Inheritance)**：不要透過繼承來獲得行為，而是透過將行為委派 (Delegate) 給其他策略物件來達成，這賦予了系統在執行時期動態抽換演算法的能力。

2. 系統架構與流程圖

    在策略模式中，我們將會看到三個主要的角色：`Client` (客戶端)、`Context` (環境/主體) 以及 `Strategy` (策略介面)。

    ```text
        (1) 客戶端決定並配置特定的策略
    [ Client (客戶端) ] -------------------------+
                                                |
                                                v
    [ Context (環境/主體) ] ----------------> [ Strategy (策略介面) ]
    + strategy: Strategy                      + executeAlgorithm()
    + setStrategy(Strategy)                              ^
    + performAction() {                                  | (3) 具體實作
        // (2) 委派給策略物件執行                          |
        strategy.executeAlgorithm();  +------------------+------------------+
    }                                 |                  |                  |
                        [ ConcreteStrategy A ] [ ConcreteStrategy B ] [ ConcreteStrategy C ]
                        (例如：輪詢負載平衡)   (例如：隨機負載平衡)   (例如：最少連線數優先)
                        + executeAlgorithm()   + executeAlgorithm()   + executeAlgorithm()
    ```

    從流程圖中可以清楚看到，`Context` 將演算法的細節外包出去了。它不知道具體的演算法是如何運作的，只知道該策略物件實作了 `Strategy` 介面，並在需要時呼叫它。

3. 總結

    1. **消滅條件判斷式：** 當你發現程式碼中充斥著一大堆 `if-else` 或 `switch` 來決定行為走向時，這通常就是引入策略模式的強烈訊號。
    2. **與狀態模式 (State Pattern) 的比較：** 在面試或架構審查時常會被問到這兩者的差異。雖然兩者的類別圖幾乎一模一樣，但**意圖不同**。策略模式通常是由「客戶端」主動配置並決定要使用哪一個演算法；而狀態模式則是物件內部隨著時間或事件，自動在不同的狀態間進行轉換。
    3. **副作用評估：** 採用策略模式的代價是系統中會多出許多代表演算法的策略物件。如果策略物件本身不需要保存狀態 (Stateless)，我們可以結合**享元模式 (Flyweight Pattern)** 讓這些策略物件變成共用的實例，以節省記憶體開銷。

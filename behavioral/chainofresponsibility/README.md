# 責任鏈模式 (Chain of Responsibility Pattern)

1. 責任鏈模式的核心概念

    **定義：** 當你希望「讓一個以上的物件有機會去處理同一個請求」時，就可以使用責任鏈模式。

    簡單來說，你會把所有負責處理請求的物件 (我們稱為 Handler) 串聯成一條「鏈」。*當請求進入系統時，會先交給鏈上的第一個物件檢查；如果它能夠處理，就直接處理掉；如果它不能處理，就會把請求往下一個物件傳遞，直到有人處理它為止。*

    流程圖輔助理解: 

        你可以把這條「鏈」想像成工廠的輸送帶或是系統中的闖關機制：

        ```text
               [ 客戶端發出請求 (Request) ]
                       |
                       v
                +-------------------+ (可以處理？) --> [ 處理請求，流程結束 ]
                |  Handler 第一關   | 
                +-------------------+
                       | (無法處理，呼叫繼任者往下傳遞)
                       v
                +-------------------+ (可以處理？) --> [ 處理請求，流程結束 ]
                |  Handler 第二關   | 
                +-------------------+
                       | (無法處理，往下傳遞)
                       v
                +-------------------+ (可以處理？) --> [ 處理請求，流程結束 ]
                |  Handler 第三關   | 
                +-------------------+
                       | (若整條鏈都無法處理)
                       v
               [ 請求未被處理 / 或由 Catch-All 機制攔截 ]
        ```

2. 應用場景與實務範例

    書中舉了一個非常生動的例子：Mighty Gumball 公司最近收到如雪片般飛來的各種電子郵件，包含垃圾信 (Spam)、粉絲信 (Fan mail)、客訴信 (Complaint) 以及要求在新地點裝設機器的請求 (New Location)。

    如果使用責任鏈模式，我們會將每封電子郵件依序交給不同的處理器 (Handler)：
    1. **SpamHandler** 先接手：判斷如果是垃圾信就直接刪除；若不是，傳給下一關。
    2. **FanHandler** 接手：如果是粉絲信就轉發給 CEO；若不是，再傳給下一關。
    3. **ComplaintHandler** 接手：如果是客訴就轉給法務部門。
    4. 以此類推...

    每個物件都在扮演處理器的角色，且每個物件都擁有指向「下一個繼任者 (successor)」的參考。

3. 優缺點與系統影響

    責任鏈模式有以下幾個重要的特性：

    主要優點 (Benefits)
    * **高度解耦 (Decoupling)：** 將發送請求的物件與接收請求的物件徹底解耦。客戶端只需把請求丟到鏈的開頭，不需要知道最後是誰處理的。
    * **簡化物件的設計：** 個別的 Handler 不需要知道整條鏈的結構，也不用保留其他所有成員的參考，它只需要認識自己的「下一個繼任者」是誰就好。
    * **動態的高彈性：** 你可以在系統執行期間，隨時動態地新增、移除鏈上的成員，或是改變它們的順序。
    * *實務應用：* 在 Windows 作業系統中，用來處理滑鼠點擊或鍵盤事件，通常就是基於這種責任鏈模式。

    主要缺點與風險 (Drawbacks)
    * **不保證請求一定會被處理：** 這是 SRE 最需要注意的地方。如果請求走到鏈的尾端，卻沒有任何一個 Handler 可以處理它，這個請求就會「掉出鏈外」而被忽略。*(實務上的解法是：我們通常會在鏈的最後面實作一個「捕捉所有 (catch-all) 的處理器」來當作兜底機制。)*
    * **除錯 (Debug) 困難：** 由於請求會在多個物件之間跳躍傳遞，在執行時期 (Runtime) 若要追蹤請求的流向、觀察狀態或進行除錯，會變得比較困難與複雜。

    總結來說，當你的系統需要根據不同條件套用多種處理邏輯，且希望保有隨時抽換、重組處理關卡的彈性時，責任鏈模式就是一個非常優雅且強大的架構選擇。

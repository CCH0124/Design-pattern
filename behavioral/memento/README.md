# 備忘錄模式 (Memento Pattern)

在架構具備「復原 (Undo)」機制、交易回滾 (Rollback) 或「系統快照 (Snapshot)」功能的系統時，我們經常會面臨一個挑戰：如何備份物件的狀態，同時又不破壞該物件的內部封裝性（Encapsulation）？

**備忘錄模式 (Memento Pattern)** 正是為此而生的經典架構。

1. 備忘錄模式的核心概念

   **定義：** 當你需要讓一個物件回復到先前的某個狀態時（例如使用者要求「復原」），就可以使用備忘錄模式。

   你可以把它想像成做虛擬機（VM）或資料庫的快照。書中舉了一個大家都很熟悉的例子：互動式角色扮演遊戲 (RPG)。當玩家好不容易過關斬將到了第 13 關，萬一不小心陣亡就得從頭來過，這絕對會讓玩家崩潰。因此，遊戲需要一個「儲存進度」的功能，讓玩家可以復活並恢復到最後一次順利過關的狀態。

   但從系統工程的角度來看，把儲存進度與狀態管理的邏輯跟遊戲核心邏輯混在一起，會讓核心物件變得異常龐大。基於單一職責原則 (Single Responsibility Principle)，備忘錄模式的解法是：**將需要儲存的狀態獨立抽取出來，放在一個叫做「備忘錄 (Memento)」的外部獨立物件中**。

   備忘錄模式有兩個首要目標：
      1. 儲存系統關鍵物件的重要狀態。
      2. 維持該關鍵物件的封裝性。

2. 系統架構與流程圖

   在備忘錄模式中，客戶端（Client）負責觸發存檔與讀檔，但它**絕對不能直接去修改或讀取備忘錄裡面的資料**。

   ```text
          (1) 請求存檔 (呼叫 getCurrentState)
   [ Client (客戶端) ] ------------------------------> [ MasterGameObject (遊戲主要物件) ]
          |                                                   | (2) 讀取內部狀態，產生備忘錄
          | (3) 取得並保管備忘錄                                 v
          | <------------------------------------------ [ GameMemento (備忘錄物件) ]
          |                                                   + savedGameState (受保護的狀態資料)
          |
          | (4) 發生意外，請求讀檔 (呼叫 restoreState)
          +-------------------------------------------> [ MasterGameObject (遊戲主要物件) ]
               (傳入步驟 3 保管的 GameMemento)                 | 
                                                              v (5) 解析備忘錄，恢復先前的狀態
   ```

   從這個架構可以看出，Client 只是個「保險箱」，它向 `MasterGameObject` 索取備忘錄，當需要恢復狀態時，再把這個備忘錄原封不動地交還給 `MasterGameObject` 的 `restoreState()` 方法。這樣一來，遊戲內部狀態的細節依然只有遊戲核心物件自己知道，完美保持了封裝性。

 3. 總結

   在開發大型系統時，我們常會評估引入該設計模式的成本與效益：

   **主要優點 (Benefits)：**
      * **保護封裝性與高內聚力：** 將已儲存的狀態從關鍵物件中抽離出來，有助於維持該物件的高內聚力 (Cohesion)。同時，關鍵物件的內部資料不會外洩給其他物件，維持了強大的封裝性。
      * **輕鬆實現復原機制：** 為系統提供了非常容易實作的狀態恢復能力 (Recovery capability)。

   **主要缺點與風險 (Drawbacks)：**
      * **效能與資源消耗：** 儲存與還原狀態可能會非常耗時。如果狀態物件非常龐大，頻繁地建立備忘錄將會帶來沉重的記憶體負擔與 CPU / IO 開銷。
      * *實務上的解法：* 如果是在 Java 系統中開發，書中建議可以考慮使用內建的「序列化 (Serialization)」機制來簡化並實作儲存系統狀態的功能。

   總結來說，當你的系統需要支援安全的狀態備份與還原機制，同時又不想把物件內部的變數攤在陽光下時，備忘錄模式會是首選的架構設計。

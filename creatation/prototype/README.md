# 原型模式 (Prototype Pattern)

在設計高併發 (High Concurrency) 系統或開發大型遊戲伺服器時，我們經常會遇到一個效能瓶頸：**頻繁地建立複雜物件會大量消耗 CPU 與記憶體資源**。如果每次產生新物件都需要繁瑣的資料庫查詢、網路請求或是複雜的運算，系統效能就會被拖垮。


1. 原型模式的核心概念

   **定義：** 當建立一個給定類別的實例（Instance）非常昂貴或複雜時，使用原型模式。它的核心精神是：**允許你透過「複製（Copy）」現有的實例來製造新的實例**。

   簡單來說，與其每次都用 `new` 關鍵字並重新跑一次複雜的建構流程，不如直接拿一個已經做好的「原型 (Prototype)」物件，把它像影印機一樣「複製 (Clone)」一份出來。在 Java 系統中，這通常意味著使用 `clone()` 方法，或是當你需要深拷貝 (Deep copy) 時使用反序列化 (Deserialization) 來完成。



2. 總結

   從系統架構的角度來看，原型模式是一把雙面刃。書中為我們總結了以下的優缺點：

   **主要優點 (Benefits)：**
   * **隱藏創建複雜度：** 對客戶端（Client）隱藏了建立新實例的複雜性。客戶端只需要拿到物件，根本不需要知道背後的具體類別是哪一個。
   * **效能優化：** 在某些情況下，直接複製 (Copy) 一個物件，比重新從頭建立 (Create) 一個新物件的效率還要高得多。
   * **動態生成：** 當系統必須在一個複雜的類別階層中，產生許多不同類型的新物件時，這個模式非常適用。

   **主要缺點與風險 (Drawbacks)：**
   * **複製機制的複雜性：** 這是使用原型模式最大的痛點。製作一個物件的複本（特別是當物件內部還包含了其他物件的參考，需要實作「深拷貝」時）有時候會變得非常複雜且難以處理。

   總結來說，當你的系統需要頻繁產生物件，且「複製現有物件的成本」遠小於「全新打造一個物件的成本」時，原型模式就是一個非常優雅且高效的架構選擇。
